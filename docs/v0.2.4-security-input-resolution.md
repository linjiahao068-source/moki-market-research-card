# V0.2.4 股票输入解析说明

## 1. V0.2.4 目标

V0.2.4 的目标是把 V0.2.3 的 ticker-only 输入升级为更通用的股票查询输入。用户可以输入美股 Ticker、港股/ A 股数字代码，或已收录的股票中文名，系统先解析为本地 mock security master 中的证券记录，再生成 Moki Market 研究卡雏形。

该版本仍然是 mock 体验验证版本，不接真实金融 API、不接数据库、不接真实 AI Agent。

## 2. 为什么从 ticker 输入升级为股票查询输入

单纯的 ticker 输入只能覆盖 `ORCL`、`NVDA`、`TSLA` 这类美股场景，无法自然支持中文用户常用的输入方式，例如：

- 港股数字代码：`00700`、`09988`、`03690`
- 省略前导 0 的港股代码：`700`、`9988`
- A 股数字代码：`600519`
- 股票中文名：`腾讯控股`、`阿里巴巴-W`、`英伟达`

Moki Market 面向中文美股和跨市场研究用户，因此输入框需要从“ticker 输入框”升级为“股票代码 / Ticker / 中文名”查询入口。

## 3. 支持的输入类型

### symbol

英文股票代码或类似 symbol 的输入，例如：

- `ORCL`
- `NVDA`
- `TSLA`
- `BRK.B`
- `BABA-W`

### numericCode

纯数字股票代码，以字符串处理，例如：

- `00700`
- `700`
- `09988`
- `9988`
- `600519`

### chineseName

已收录在 mock security master 中的中文名或中文别名，例如：

- `腾讯控股`
- `阿里巴巴-W`
- `美团-W`
- `小米集团-W`
- `甲骨文`
- `英伟达`
- `特斯拉`

中文名只做精确匹配，不做模糊搜索。

## 4. SecurityRecord 结构说明

`SecurityRecord` 位于：

```text
src/types/security.ts
```

核心字段包括：

- `id`：本地唯一 ID
- `market`：市场类型，支持 `US`、`HK`、`CN_A`、`UNKNOWN`
- `symbol`：英文 symbol，例如 `ORCL`、`NVDA`、`0700.HK`
- `numericCode`：数字代码，例如 `00700`、`09988`、`600519`
- `codeVariants`：数字代码变体，例如 `700`、`0700`、`00700`
- `companyName`：公司英文或通用显示名称
- `englishName`：英文名称
- `chineseNameHK`：港股或中文语境下常见中文名
- `chineseAliases`：中文别名，必须精确匹配
- `theme`：mock 研究主题，用于生成研究卡文案

## 5. SecurityResolution 三种状态

`SecurityResolution` 是 `resolveSecurityInput` 的返回类型，支持三种状态。

### matched

表示输入成功匹配到唯一证券。

包含：

- `status: "matched"`
- `inputKind`
- `matchType`
- `rawInput`
- `normalizedInput`
- `security`

### unmatched

表示输入格式可接受，但没有匹配到 mock 主数据。

此时会生成 `fallbackSecurity`，用于创建通用 mock 研究卡。

包含：

- `status: "unmatched"`
- `inputKind`
- `rawInput`
- `normalizedInput`
- `fallbackSecurity`

### ambiguous

表示输入匹配到多个候选证券。

此时不能默认选第一个，需要展示候选项，让用户选择更精确的股票。

包含：

- `status: "ambiguous"`
- `inputKind`
- `rawInput`
- `normalizedInput`
- `candidates`

## 6. mockSecurities 的作用

`mockSecurities` 位于：

```text
src/data/mockSecurities.ts
```

它是 V0.2.4 的本地 mock security master，负责提供股票输入解析所需的主数据。

它用于：

- symbol 精确匹配
- numericCode 精确匹配
- codeVariants 匹配
- chineseNameHK 精确匹配
- chineseAliases 精确匹配
- 为 mock 研究卡提供 companyName、market、theme 等基础信息

后续接入真实证券主数据时，应优先替换这个数据层或替换 resolver 的数据源，而不是改动 UI。

## 7. 为什么 numericCode 必须是 string

股票数字代码不是普通数字，而是证券标识符。

例如港股：

```text
00700
09988
03690
```

如果转成 Number：

```text
00700 -> 700
09988 -> 9988
03690 -> 3690
```

会丢失前导 0，破坏证券代码的展示和匹配。因此 `numericCode` 必须始终使用 string，不能使用 Number，也不要用 `parseInt` 或 `Number()` 处理。

## 8. 为什么 V0.2.4 不做模糊搜索

V0.2.4 不做模糊搜索，原因是：

- 证券识别需要可解释、可复盘。
- 中文名模糊匹配容易误识别证券。
- 当前没有真实证券主数据和同名消歧能力。
- Moki Market 是研究辅助工具，不应在证券身份上做 AI 猜测。

因此中文名只做精确匹配：

- 匹配 `chineseNameHK`
- 匹配 `chineseAliases`
- 不做 `includes`
- 不做拼音转换
- 不做 fuzzy matching

## 9. 为什么 V0.2.4 不接真实 API

V0.2.4 是产品体验和输入解析验证阶段，重点是验证：

- 输入框是否能支持多种股票输入方式
- security resolver 是否能稳定输出 matched / unmatched / ambiguous
- 生成页是否能根据解析结果展示研究卡雏形
- UI 是否适合中文投资者使用

因此暂不接真实 API，避免引入：

- 数据授权问题
- API key 管理
- 网络错误处理
- 数据新鲜度问题
- 合规和责任边界问题

当前结果均为 mock，用于产品体验验证，不构成投资建议。

## 10. V0.2.5 如何替换为真实证券主数据

V0.2.5 可以在不重写 UI 的前提下替换数据源：

1. 保留 `SecurityRecord` 和 `SecurityResolution` 类型。
2. 保留 `resolveSecurityInput(rawInput)` 的对外接口。
3. 将 `mockSecurities` 替换为真实证券主数据源，或在 resolver 内部切换到真实数据查询。
4. 增加真实数据的 symbol、numericCode、中文名、英文名、交易所、货币等字段。
5. 保持中文名精确匹配和 ambiguous 候选机制。
6. 只有在数据质量和合规边界明确后，再考虑模糊搜索或拼音搜索。

推荐迁移方向：

```text
UI 输入
  -> resolveSecurityInput(rawInput)
  -> 真实 security master
  -> SecurityResolution
  -> mock / real research card generator
```

这样可以保持 V0.2.4 的 UI 和生成流程稳定，只替换底层数据来源。
