# V0.2.5 真实基础数据接入 MVP

## 1. V0.2.5 目标

V0.2.5 的目标是在 V0.2.4 股票输入解析能力之上，新增真实基础数据接入层。用户在 `/generate` 输入股票代码 / Ticker / 中文名后，系统先解析证券身份，再尝试获取基础公司和财务数据，最后生成 Moki Market 研究卡雏形。

该版本仍然是 MVP：真实数据只用于基础资料和财务字段补充，研究卡结构仍由 Moki Market mock generator 生成，不构成投资建议。

## 2. 为什么先接 SEC EDGAR

SEC EDGAR 是美国上市公司公告和 XBRL 基础财务数据的重要公开来源，适合 V0.2.5 的 MVP 目标：

- 覆盖美股公司。
- 可通过 CIK / ticker 映射找到公司。
- 可获取 submissions、filings 和 companyfacts。
- 数据来源清晰，适合展示 sourceLinks。
- 不需要先引入数据库。

因此 V0.2.5 优先使用 SEC EDGAR 作为美股基础数据来源。

## 3. SEC Provider 支持范围

当前 SEC Provider 设计用于：

- `market === "US"` 且有 `symbol` 的证券。
- 或者直接提供 `cik` 的证券。

主要能力：

- 通过 `company_tickers.json` 查 CIK。
- 读取 `submissions/CIK##########.json`。
- 提取公司名、ticker、exchange、最近的 `10-K` / `10-Q` / `8-K`。
- 读取 `companyfacts/CIK##########.json`。
- 保守提取 revenue、net income、assets、liabilities、cash、operating income、EPS 等字段。

限制：

- 不覆盖港股和 A 股。
- 不做复杂财务模型。
- 不做同比 / 环比计算。
- 字段缺失时使用 warnings 告知。

## 4. FMP Provider 可选范围

FMP Provider 是可选补充数据源。

启用条件：

```env
FMP_API_KEY="your-key"
```

如果没有配置 `FMP_API_KEY`，系统会跳过 FMP Provider。

当前尝试请求：

- `/profile?symbol={symbol}`
- `/quote?symbol={symbol}`
- `/income-statement?symbol={symbol}`

映射字段包括：

- company profile
- quote
- income statement 的部分 financials 字段

FMP Provider 不应把 API key 暴露到浏览器，所有请求应在 server-side provider 层完成。

## 5. Mock fallback 机制

V0.2.5 使用 fallback-first 思路保障页面稳定。

以下情况会 fallback 到 mock：

- SEC 不支持该 market。
- SEC 找不到 CIK。
- SEC 请求失败。
- SEC 返回数据字段不足。
- FMP 未配置 API key。
- FMP 请求失败。
- Provider 超时或返回格式异常。

fallback 后仍会生成研究卡雏形，并在 warnings 或 sourceNote 中提示：

```text
当前使用 mock basic data fallback，不代表真实财务数据。
```

港股 / A 股在 V0.2.5 暂时使用 mock fallback，并提示：

```text
V0.2.5 暂未接入该市场真实基础数据。
```

## 6. BasicCompanyData 数据结构

`BasicCompanyData` 位于：

```text
src/types/basic-data.ts
```

核心字段：

- `provider`：数据来源，例如 `sec-edgar`、`fmp`、`mock`
- `coverageStatus`：覆盖状态，支持 `full`、`partial`、`empty`、`failed`
- `freshnessStatus`：新鲜度，支持 `fresh`、`stale`、`unknown`
- `fetchedAt`：获取时间
- `security`：解析后的证券记录
- `profile`：公司基础信息
- `latestFiling`：最近 filing
- `financials`：基础财务字段，全部可选
- `quote`：行情字段，全部可选
- `sourceLinks`：来源链接数组
- `warnings`：数据缺失或 fallback 提示数组

设计原则：字段缺失时不应让页面报错，UI 应展示 `--` 或温和提示。

## 7. 为什么 API key 不能暴露到浏览器

真实数据 Provider 可能需要 API key，例如 FMP。

API key 不能暴露到浏览器，原因：

- 前端 bundle 可以被用户查看。
- `NEXT_PUBLIC_` 变量会被打包到浏览器端。
- API key 暴露后可能被滥用。
- 供应商通常要求服务端安全调用。

因此：

- `FMP_API_KEY` 只能在 server-side provider 中读取。
- 不使用 `NEXT_PUBLIC_FMP_API_KEY`。
- `/generate` 前端通过 API route 请求服务端，由服务端访问 Provider。

## 8. static export 下 API route 的限制

当前项目没有启用：

```ts
output: 'export'
```

因此可以使用 Next.js API route。

如果未来启用 static export：

```ts
output: 'export'
```

则：

- `app/api/basic-data/route.ts` 不能被静态导出。
- `/api/basic-data` 不可用。
- 服务端读取 `process.env` 的 Provider 逻辑也无法运行。

静态导出时的替代方案：

1. 继续使用纯前端 mock 数据。
2. 把真实数据接入迁移到外部 API 服务。
3. 取消 static export，使用支持 server/serverless runtime 的部署方式。

## 9. V0.2.6 如何接新闻和舆情

V0.2.6 可以在 V0.2.5 基础数据层之上继续扩展新闻和舆情输入。

建议方向：

- 新增 `NewsProvider` 接口，负责公司相关新闻和公告摘要。
- 新增 `SentimentProvider` 接口，负责 X / TikTok / 社交平台讨论线索。
- 数据结构仍保持 provider-agnostic，避免绑定具体供应商。
- 新闻和舆情只作为研究线索，不生成操作建议。
- 继续保留 fallback 机制，Provider 失败时仍能生成 mock 研究卡雏形。

推荐后续链路：

```text
SecurityResolution
  -> BasicCompanyData
  -> NewsSignals
  -> SentimentSignals
  -> ResearchCard
```

在接入新闻和舆情前，应先明确：

- 来源授权
- 缓存策略
- 速率限制
- 合规边界
- 非投资建议提示
